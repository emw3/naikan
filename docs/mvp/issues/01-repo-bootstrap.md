## Agent Brief

**Category:** enhancement
**Summary:** Bootstrap the empty repo into a runnable Bun workspaces scaffold — Hono API + Svelte SPA + raw-SQL Postgres + migrations runner + a plain `/health` + a CI smoke test.

**Current behavior:**
The repo is empty — no `package.json`, no source, nothing builds or runs.

**Desired behavior:**
`bun dev` starts the Hono API and serves the Svelte SPA (placeholder home page at `/`); `curl /health` returns `200 OK`; `bun migrate` applies an empty baseline migration against a local Postgres and `bun migrate:down` rolls it back; a portable smoke script and a GitHub Actions workflow verify install → build → start → `GET /health` → 200.

**Decisions resolved in triage grilling (these are the contract):**

- **Layout — Bun workspaces monorepo, per ADR-0005.** `packages/*` = plain-TypeScript kernel modules with **no Bun-specific APIs**; `apps/*` = `api` (Bun + Hono), `worker` (minimal placeholder — runtime Bun-vs-Node is decided later in #02), `web-admin` (Svelte 5 + Vite). The plain-TS kernel split is load-bearing: it preserves the Node-worker fallback. Do not contaminate `packages/*` with Bun-only APIs.
- **Migrations — `node-pg-migrate`, raw SQL, no ORM.** Up/down migrations; scripts `bun migrate` (up) and `bun migrate:down` (down). The baseline migration is empty (establishes the migration-tracking table + a no-op forward/rollback).
- **DB access — raw SQL via the `postgres` library**, connection string from env (`DATABASE_URL`). No ORM (resolves the PRD's "Drizzle-equivalent" wording in favor of PRD line 74's "no ORM").
- **`/health` — returns `200 OK` plain.** No queue or freshness assertions yet — those land in #18.
- **CI — portable script + workflow.** A `bun run smoke` script holds the real logic (install → build → start the API → poll `/health` → exit non-zero if not 200); `.github/workflows/ci.yml` simply invokes `bun run smoke`. Repo targets GitHub.
- **Svelte 5.**
